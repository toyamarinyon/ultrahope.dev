### 導入: 人間がAIエージェントに入力した「意図」をレビューしたいと思っていた

コーディングエージェントを使うようになり「意図」が見えづらくなった
どういうことかというと、コーディングエージェントを使うと、最初に作りたいものを人間がコーディングエージェントに伝えます。これは良いです。この時inputは意図でoutputがコードです。
1. 人間→コーディングエージェントに実装を依頼（input: 意図）→コード

次が問題で、コードを確認して問題なければ（確認しない人も少なくないと思いますが）コーディングエージェントにPull requestを作ってもらいます。この時inputはコードで、outputがpull requestです。コーディングエージェントによってはここでも意図が含まれる場合もありますが、分断されることもあり、その時は以下のようになります。
2. 人間→コーディングエージェントにPull request作成を依頼（input: コード）→ pull request
最初の意図が消失し、コードからPull requestのtitleやdescriptionを生成します。

そのtitleやdescriptionは伝言ゲームのように最初の意図が何かしらの形で湾曲しています。こうなるとレビューが難しいですね。
（メモ）当初のintentが消失した状態で、コードから意図を推論したとき、元のintentを復元できる可能性は高くないということを説明したい。

そこで、Coding Agentのhook機能を使って、ユーザーがCoding Agentに送信したPromptをファイルに残しておき、Pull requestを作るときにそれを参照すれば複数のセッションにまたがった開発になってもそれぞれの人間の意図を考慮したPull requestを作成できるのではないか、と思い試してこれはうまくいきました。
ただ、個人ではうまくいったのですが、これをチームの開発で使うにはチームメンバーそれぞれが使っているCoding Agentのhook機能のフォーマットに合わせるアダプター層を作る必要があり、これは時間がかかるなと後回しにしていました。

そんな中知ったのが entire でした。そうそうこれこれという感じでした。

### やってみた

entireにはcurlでインストールできますが、これだとグローバルに入るので、ちょっと怖い。`go install`でもインストールできるので、これでやりました。goはmiseで管理しているのでその中に入ります。

私はCodexを使っているのでインストール後はentireを使いたいレポジトリで

```bash
entire agent add codex
```
でcodex用のhooksや設定ファイルが作成されます。

注意点としては、Codexはhookがデフォルトで無効になっており、Codex CLIで`/hooks`コマンドを使ってApproveしないと有効になりません。Codex CLIだと起動後にテキストでその旨が説明されるので気付けますが、Codex Appだと特に何も出ないのでまずはCodex CLIで試すのがおすすめです。



hooksが有効になると、その後は特に作業は必要なく、これまで通りCodex CLIやCodex Appを使うとhookで適宜情報が保存され、や push のタイミングで entire.io にデータが送信されます。


```参考情報
hook が「ローカルに保存した内容」を外へ送るタイミングは **Git の `pre-push` hook 実行時**です。つまり通常は、ユーザーが `git push` した直前に、checkpoint/session log 用の refs/branch も一緒に push されます。

流れはこうです。

1. AI agent の turn 終了時  
   `handleLifecycleTurnEnd` が transcript / prompt / 変更ファイル情報をローカルに保存し、checkpoint を作ります。ここではまだリモート送信ではありません。  
   参照: [lifecycle.go#L434-L438](https://github.com/entireio/cli/blob/30e58ea38ee7315a3477045570291c1415f94f01/cmd/entire/cli/lifecycle.go#L434-L438), [lifecycle.go#L532-L620](https://github.com/entireio/cli/blob/30e58ea38ee7315a3477045570291c1415f94f01/cmd/entire/cli/lifecycle.go#L532-L620)

2. `git commit` 後  
   `post-commit` hook が動き、`Entire-Checkpoint` trailer 付き commit を見て、shadow branch の内容を checkpoint として「condense」します。これも基本はローカル Git refs への整理です。  
   参照: [manual_commit_hooks.go#L635-L645](https://github.com/entireio/cli/blob/30e58ea38ee7315a3477045570291c1415f94f01/cmd/entire/cli/strategy/manual_commit_hooks.go#L635-L645), [manual_commit_hooks.go#L875-L905](https://github.com/entireio/cli/blob/30e58ea38ee7315a3477045570291c1415f94f01/cmd/entire/cli/strategy/manual_commit_hooks.go#L875-L905)

3. `git push` 直前  
   `pre-push` hook が `entire hooks git pre-push "$1"` を実行します。コメントにも “push session logs alongside user's push” とあり、ここが外へ出るタイミングです。  
   参照: [hooks.go#L202-L208](https://github.com/entireio/cli/blob/30e58ea38ee7315a3477045570291c1415f94f01/cmd/entire/cli/strategy/hooks.go#L202-L208)

実際の送信処理は `ManualCommitStrategy.PrePush` で、`entire/checkpoints/v1` branch と、設定次第で v2 refs を push します。`checkpoint_remote` が設定されていればユーザーの push 先ではなく別 repo に push されます。`push_sessions: false` なら送信しません。  
参照: [manual_commit_push.go#L11-L24](https://github.com/entireio/cli/blob/30e58ea38ee7315a3477045570291c1415f94f01/cmd/entire/cli/strategy/manual_commit_push.go#L11-L24), [manual_commit_push.go#L25-L50](https://github.com/entireio/cli/blob/30e58ea38ee7315a3477045570291c1415f94f01/cmd/entire/cli/strategy/manual_commit_push.go#L25-L50)

短く言うと、**hook でローカル保存は turn end / post-commit で進み、entire.io 側から見える場所への送信は `git push` 時の `pre-push` hook**です。CLI が毎 turn ごとに直接 entire.io API へアップロードしている、という実装には見えません。
```

### entire/checkpoints/v1 ブランチについて

entire agent add するとhookと合わせて `entire/checkpoints/v1`ブランチが作成されます。

これは、以下のようなものなのでいい感じに説明してください。


```
`entire/checkpoints/v1` は、Entire CLI が作る **専用の Git ブランチ名**です。

意味としては、「通常の `main` などの開発ブランチとは別に、AI セッションの checkpoint / transcript / prompt / metadata を置くためのメタデータ用ブランチ」です。コード上では `MetadataBranchName` として定義されています。

```go
const MetadataBranchName = "entire/checkpoints/v1"
```

このブランチには、たとえば次のようなものが入ります。

- checkpoint metadata
- session transcript
- user prompt
- touched files
- token usage
- commit との対応情報

保存形式は checkpoint ID で shard されます。例として checkpoint ID が `a3b2c4d5e6f7` なら:

```text
entire/checkpoints/v1
└── a3/
    └── b2c4d5e6f7/
        ├── metadata.json
        └── 0/
            ├── full.jsonl
            ├── prompt.txt
            └── content_hash.txt
```

通常のユーザー commit には、次のような trailer が入ります。

```text
Entire-Checkpoint: a3b2c4d5e6f7
```

この ID を使って、

```text
main の commit
  -> Entire-Checkpoint ID
  -> entire/checkpoints/v1 上の metadata/transcript
```

という対応を引ける、という仕組みです。

大事なのは、`entire/checkpoints/v1` は **作業ブランチではなく、メタデータ保管用ブランチ**という点です。`post-commit` hook でローカルにこのブランチへ整理され、`pre-push` hook でユーザーの `git push` に合わせて push されます。

また privacy 的にはかなり重要で、公式 docs にも「session transcripts, user prompts, checkpoint metadata are committed to `entire/checkpoints/v1`」とあります。つまり remote に push されると、リポジトリにアクセスできる人はこのブランチ上の transcript / prompt / metadata を見られます。

補足すると、今の実装には v2 用の refs もあります。

```text
refs/entire/checkpoints/v2/main
refs/entire/checkpoints/v2/full/current
```

なので `entire/checkpoints/v1` は名前通り **v1 の checkpoint metadata branch**で、設定によっては v2 refs 側を使う/併用する構造になっています。
```
