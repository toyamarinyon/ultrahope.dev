+title: FrontierCodeの評価設計から学ぶ、コーディングエージェントに依頼するときの心がけ

+
+コーディングエージェントに実装を依頼するとき、最近は「何を作ってほしいか」だけでは足りないのだろうな、と思うことが増えました。
+
+もちろん、目的を伝えることは大切です。
+
+このバグを直してほしい。この画面を作ってほしい。このリファクタリングを進めてほしい。そう頼めば、エージェントはかなりのところまで進んでくれます。テストを走らせ、エラーを読み、差分を作り、説明まで添えてくれます。
+
+それでも、差分を眺めながら少し立ち止まることがあります。
+
+たしかに動いている。テストも通っている。けれど、このPRを自分は本当に受け入れたいのだろうか。
+
+これは、単にエージェントの性能が足りないという話ではない気がします。むしろ、エージェントがかなり実装できるようになったからこそ、こちらが何をよい変更だと思っているのかが前に出てくる。

+### テストが通ることの、その先
+
+これまでコーディングエージェントの評価としてよく参照されてきたものに、SWE-benchがあります。SWE-benchは、GitHub上の実際のissueとコードベースをもとに、モデルが問題を解決するpatchを生成できるかを見るベンチマークです。[^swe-bench]
+

+
+これはとても重要な評価です。
+
+実際のコードベースを読み、issueを理解し、必要な変更を入れ、テストを通す。HumanEvalのような小さな関数を書く評価から比べると、かなり実務に近い場所へ踏み込んでいます。
+
+ただ、実務のコードレビューでは、テストが通っただけでは終わりません。
+
+変更範囲は妥当か。既存の抽象に沿っているか。テストは本当に今回の振る舞いを守っているか。名前や責務はこのリポジトリの文脈に合っているか。将来ここを変更するとき、次の人が追える形になっているか。
+
+こういう問いは、テストの緑色だけでは拾いきれません。
+
+FrontierCodeが面白いのは、ここを正面から評価しようとしているところです。記事では、これから問うべきなのは「モデルは正しいコードを書けるか」だけではなく、「モデルはよいコードを書けるか」だとされています。
+
+もう少し実務の言葉に寄せるなら、問いはこうです。
+
+このPRを、メンテナは本当にmergeするだろうか。
+
+### メンテナの判断を評価に入れる
+
+FrontierCodeでは、20人以上のOSSメンテナが、36の主要OSSリポジトリからタスクを作っています。各タスクには40時間以上をかけ、メンテナ自身がそのリポジトリにおける「mergeできる変更」の基準を評価項目に落とし込んだ、と説明されています。
+
+評価軸には、behavioral correctness、regression safety、mechanical cleanliness、test correctness、scope、code qualityが並びます。
+
+挙動が正しいか。既存の挙動を壊していないか。build、lint、style checkが通るか。追加したテストが意味を持っているか。必要な範囲だけを触っているか。コードが既存の設計や慣習に合っていて、共同作業者にとって読めるものになっているか。
+
+この並びを見たとき、これはベンチマークの設計であると同時に、かなりそのままコードレビューの観点だと思いました。
+
+FrontierCodeでは、criteriaをblockerとnon-blockerに分けます。blockerは、メンテナがコードレビューでhard stopにするような条件です。たとえば、仕様を満たしていない、重要な性能要件を壊している、変更範囲が大きすぎる、といったものです。non-blockerは、style、type safety、readabilityのように、品質スコアには効くけれど、それ単独では必ずしもmergeを止めないものとして扱われます。
+
+この分け方は、エージェントに依頼するときにもかなり効きそうです。
+
+人間がレビューコメントを書くとき、すべての指摘が同じ重さになってしまうことがあります。「これは絶対に直してほしい」と「ここは好みだけれど、できれば整えたい」が同じ口調で並ぶ。人間同士でも少し大変ですが、エージェントに対してはなおさらです。
+
+どれが成功条件なのか。
+どれが品質の好みなのか。
+どれは今回触らなくてよいのか。
+
+そこを分けて渡すだけで、依頼の質はかなり変わる気がします。
+
+### テストを書いたことを、テストし直す
+
+FrontierCodeの記事で特に印象に残ったのは、reverse-classicalという評価方法でした。
+
+これは、エージェントが追加したテストを、修正前の壊れたコードベースに対して実行し、ちゃんと失敗するかを見るものです。修正前でも通ってしまうテストなら、そのテストは今回の問題を捕まえていないかもしれません。
+
+言われてみれば当たり前です。
+
+でも、エージェントに実装を任せていると、「テストも追加しました」という報告に少し安心してしまうことがあります。差分を見るとテストファイルが増えている。CIも通っている。説明ももっともらしい。それだけで、なんとなく守られた気持ちになる。
+
+けれど、そのテストが本当に壊れていた振る舞いを固定しているかは、別の問いです。
+
+これは自分の作業にもそのまま持ち込めます。
+
+エージェントにバグ修正を頼むなら、最初から「このテストは修正前には落ち、修正後に通ることを確認してください」と伝える。あるいは、こちらがレビューするときに、追加テストが何を捕まえているのかを一緒に確認する。
+
+テストを書くことと、安心できることの間には距離があります。
+
+その距離を、エージェント任せにしすぎない。ここはかなり大事だと思いました。
+
+### 触ってよい範囲を決める
+
+もう一つ、自分の実感に近かったのはscopeの評価です。
+
+エージェントは、ときどき親切です。
+
+頼んだ修正のついでに、近くの命名を整えたり、少し広めの抽象化をしたり、関連しそうなファイルまで触ったりします。それが本当に助かることもあります。人間なら後回しにしていた小さな掃除を、まとめて片づけてくれることもある。
+
+ただ、いつも歓迎できるわけではありません。
+
+小さなバグ修正のつもりだったのに、差分が広がる。レビューする範囲が増える。何が今回の目的に必要な変更で、何がついでの整理なのかが混ざる。すると、テストが通っていても受け入れにくくなります。
+
+FrontierCodeのscope criterionは、file boundary、diff size、semantic localityのような観点で、変更が必要な範囲に収まっているかを見ます。これは、エージェントへの依頼にもほとんどそのまま使えます。
+
+「この修正はこのmodule内に収めたい」
+「public APIは変えない」
+「今回はformattingだけの差分を混ぜない」
+「既存の抽象を使い、別の抽象は足さない」
+「必要なら先に相談してから範囲を広げる」
+
+こういう制約は、エージェントを縛るためだけのものではありません。
+
+むしろ、レビューしやすく、受け入れやすく、あとから思い出せる変更にするための足場です。
+
+### 依頼者が小さなメンテナになる
+
+FrontierCodeの記事には、`LOG_WARNING()`という警告ログ用のhelperを追加する例が出てきます。
+
+あるモデルの解法は、挙動としては正しく見えます。複数行のwarningも標準エラーに出ます。ただし、1行目だけ`LOG_WARNING()`を使い、続く行では`std::cerr`を直接使っていました。今はどちらも同じstreamに出るので動きます。けれど、将来`LOG_WARNING()`の実装が変わったとき、call site側に「これは`std::cerr`と同じだ」という前提が残ってしまう。
+
+これは、テストでは通るけれど、メンテナとしては気になる変更です。
+
+抽象を作ったなら、その抽象を使い切ってほしい。既存の設計意図があるなら、その意図に沿ってほしい。今たまたま動くことだけではなく、次に触る人がどの前提で読めばよいのかを残してほしい。
+
+この感覚は、コーディングエージェントと一緒に作業するときにも必要になります。
+
+エージェントに「作って」と頼むだけなら、こちらは利用者に近い立場です。けれど、出てきた差分を受け入れ、将来もそのコードを育てるなら、こちらは少しメンテナになります。
+
+大きなOSSのメンテナでなくても、自分のリポジトリには自分なりの文脈があります。
+
+どこまでを今の変更に含めるのか。
+どの抽象を使うのか。
+どのテストで守るのか。
+どの違和感は今直し、どの違和感は別のissueに分けるのか。
+
+こうした判断は、エージェントが完全には肩代わりできません。むしろ、エージェントが実装できる範囲が広がるほど、人間の側に残る判断が見えやすくなります。
+
+### 依頼の前に置いておきたいもの
+
+最近、エージェントに依頼するときは、いきなり実装内容を書き始めるより、先に小さなレビュー基準を置いたほうがよいのだと思うようになりました。
+
+たとえば、これくらいで十分です。
+
+```text
+目的:
+成功条件:
+触ってよい範囲:
+触ってほしくない範囲:
+追加/更新すべきテスト:
+互換性で守ること:
+既存パターンで従うべきもの:
+完了前に走らせるコマンド:
+```
+
+これは、重たい仕様書ではありません。
+
+むしろ、今から作る差分をどう受け入れるかを、先に少しだけ言葉にしておくものです。すべてを決めきる必要はありません。途中でわからないことが出てきたら、そこで相談すればよい。
+
+大事なのは、何をもって「よい変更」とするのかを、完全にエージェントの出力後に任せないことです。
+
+FrontierCodeの評価設計から学べるのは、ベンチマークの作り方だけではないと思います。
+
+テストが通ること。
+変更範囲が抑えられていること。
+追加したテストが本当に意味を持つこと。
+既存の設計に沿っていること。
+blockerとnon-blockerを分けること。
+
+そういう観点を、私たち自身の依頼にも少し持ち込める。
+
+コーディングエージェントにうまく書かせる、というより、エージェントが出してきた変更を自分たちが受け入れられる形に近づけていく。
+
+そのために、依頼者は少しだけメンテナになる。
+
+私はしばらく、この感覚を手元に置いておきたいです。エージェントに任せる部分を増やしながら、それでも何をmergeしたいのか、どんな変更なら育てていけるのかは、自分の側にも残しておく。
+
+速く作れることと、受け入れられる形で作れることは、少し違います。
+
+その違いを忘れないための手がかりとして、FrontierCodeの評価設計はとてもよいものに見えました。

+### テストが通ることの、その先
+
+これまでコーディングエージェントの評価としてよく参照されてきたものに、SWE-benchがあります。SWE-benchは、GitHub上の実際のissueとコードベースをもとに、モデルが問題を解決するpatchを生成できるかを見るベンチマークです。[^swe-bench]

+
+実際のコードベースを読み、issueを理解し、必要な変更を入れ、テストを通す。HumanEvalのような小さな関数を書く評価から比べると、かなり実務に近い場所へ踏み込んでいます。

+publishedAt: "2026-06-15"

+コーディングエージェントに実装を依頼するとき、最近は「何を作ってほしいか」だけでは足りないのだろうな、と思うことが増えました。
+
+もちろん、目的を伝えることは大切です。
+
+このバグを直してほしい。この画面を作ってほしい。このリファクタリングを進めてほしい。そう頼めば、エージェントはかなりのところまで進んでくれます。テストを走らせ、エラーを読み、差分を作り、説明まで添えてくれます。
+
+それでも、差分を眺めながら少し立ち止まることがあります。
+
+たしかに動いている。テストも通っている。けれど、このPRを自分は本当に受け入れたいのだろうか。
+
+これは、単にエージェントの性能が足りないという話ではない気がします。むしろ、エージェントがかなり実装できるようになったからこそ、こちらが何をよい変更だと思っているのかが前に出てくる。
+
+そんなことを考えていたときに、Cognitionが公開した[FrontierCode](https://cognition.ai/blog/frontier-code)の記事を読みました。
+
+### テストが通ることの、その先
+
+これまでコーディングエージェントの評価としてよく参照されてきたものに、SWE-benchがあります。SWE-benchは、GitHub上の実際のissueとコードベースをもとに、モデルが問題を解決するpatchを生成できるかを見るベンチマークです。[^swe-bench]
+
+OpenAIが公開したSWE-bench Verifiedの記事では、SWE-benchの各サンプルには、修正前には失敗し修正後には通る`FAIL_TO_PASS` testと、既存機能を壊していないかを見る`PASS_TO_PASS` testがあると説明されています。[^swe-bench-verified]
+
+これはとても重要な評価です。
+
+実際のコードベースを読み、issueを理解し、必要な変更を入れ、テストを通す。HumanEvalのような小さな関数を書く評価から比べると、かなり実務に近い場所へ踏み込んでいます。
+
+ただ、実務のコードレビューでは、テストが通っただけでは終わりません。
+
+変更範囲は妥当か。既存の抽象に沿っているか。テストは本当に今回の振る舞いを守っているか。名前や責務はこのリポジトリの文脈に合っているか。将来ここを変更するとき、次の人が追える形になっているか。
+
+こういう問いは、テストの緑色だけでは拾いきれません。
+
+FrontierCodeが面白いのは、ここを正面から評価しようとしているところです。記事では、これから問うべきなのは「モデルは正しいコードを書けるか」だけではなく、「モデルはよいコードを書けるか」だとされています。
+
+もう少し実務の言葉に寄せるなら、問いはこうです。
+
+このPRを、メンテナは本当にmergeするだろうか。
+
+### メンテナの判断を評価に入れる
+
+FrontierCodeでは、20人以上のOSSメンテナが、36の主要OSSリポジトリからタスクを作っています。各タスクには40時間以上をかけ、メンテナ自身がそのリポジトリにおける「mergeできる変更」の基準を評価項目に落とし込んだ、と説明されています。
+
+評価軸には、behavioral correctness、regression safety、mechanical cleanliness、test correctness、scope、code qualityが並びます。
+
+挙動が正しいか。既存の挙動を壊していないか。build、lint、style checkが通るか。追加したテストが意味を持っているか。必要な範囲だけを触っているか。コードが既存の設計や慣習に合っていて、共同作業者にとって読めるものになっているか。
+
+この並びを見たとき、これはベンチマークの設計であると同時に、かなりそのままコードレビューの観点だと思いました。
+
+FrontierCodeでは、criteriaをblockerとnon-blockerに分けます。blockerは、メンテナがコードレビューでhard stopにするような条件です。たとえば、仕様を満たしていない、重要な性能要件を壊している、変更範囲が大きすぎる、といったものです。non-blockerは、style、type safety、readabilityのように、品質スコアには効くけれど、それ単独では必ずしもmergeを止めないものとして扱われます。
+
+この分け方は、エージェントに依頼するときにもかなり効きそうです。
+
+人間がレビューコメントを書くとき、すべての指摘が同じ重さになってしまうことがあります。「これは絶対に直してほしい」と「ここは好みだけれど、できれば整えたい」が同じ口調で並ぶ。人間同士でも少し大変ですが、エージェントに対してはなおさらです。
+
+どれが成功条件なのか。
+どれが品質の好みなのか。
+どれは今回触らなくてよいのか。
+
+そこを分けて渡すだけで、依頼の質はかなり変わる気がします。
+
+### テストを書いたことを、テストし直す
+
+FrontierCodeの記事で特に印象に残ったのは、reverse-classicalという評価方法でした。
+
+これは、エージェントが追加したテストを、修正前の壊れたコードベースに対して実行し、ちゃんと失敗するかを見るものです。修正前でも通ってしまうテストなら、そのテストは今回の問題を捕まえていないかもしれません。
+
+言われてみれば当たり前です。
+
+でも、エージェントに実装を任せていると、「テストも追加しました」という報告に少し安心してしまうことがあります。差分を見るとテストファイルが増えている。CIも通っている。説明ももっともらしい。それだけで、なんとなく守られた気持ちになる。
+
+けれど、そのテストが本当に壊れていた振る舞いを固定しているかは、別の問いです。
+
+これは自分の作業にもそのまま持ち込めます。
+
+エージェントにバグ修正を頼むなら、最初から「このテストは修正前には落ち、修正後に通ることを確認してください」と伝える。あるいは、こちらがレビューするときに、追加テストが何を捕まえているのかを一緒に確認する。
+
+テストを書くことと、安心できることの間には距離があります。
+
+その距離を、エージェント任せにしすぎない。ここはかなり大事だと思いました。
+
+### 触ってよい範囲を決める
+
+もう一つ、自分の実感に近かったのはscopeの評価です。
+
+エージェントは、ときどき親切です。
+
+頼んだ修正のついでに、近くの命名を整えたり、少し広めの抽象化をしたり、関連しそうなファイルまで触ったりします。それが本当に助かることもあります。人間なら後回しにしていた小さな掃除を、まとめて片づけてくれることもある。
+
+ただ、いつも歓迎できるわけではありません。
+
+小さなバグ修正のつもりだったのに、差分が広がる。レビューする範囲が増える。何が今回の目的に必要な変更で、何がついでの整理なのかが混ざる。すると、テストが通っていても受け入れにくくなります。
+
+FrontierCodeのscope criterionは、file boundary、diff size、semantic localityのような観点で、変更が必要な範囲に収まっているかを見ます。これは、エージェントへの依頼にもほとんどそのまま使えます。
+
+「この修正はこのmodule内に収めたい」
+「public APIは変えない」
+「今回はformattingだけの差分を混ぜない」
+「既存の抽象を使い、別の抽象は足さない」
+「必要なら先に相談してから範囲を広げる」
+
+こういう制約は、エージェントを縛るためだけのものではありません。
+
+むしろ、レビューしやすく、受け入れやすく、あとから思い出せる変更にするための足場です。
+
+### 依頼者が小さなメンテナになる
+
+FrontierCodeの記事には、`LOG_WARNING()`という警告ログ用のhelperを追加する例が出てきます。
+
+あるモデルの解法は、挙動としては正しく見えます。複数行のwarningも標準エラーに出ます。ただし、1行目だけ`LOG_WARNING()`を使い、続く行では`std::cerr`を直接使っていました。今はどちらも同じstreamに出るので動きます。けれど、将来`LOG_WARNING()`の実装が変わったとき、call site側に「これは`std::cerr`と同じだ」という前提が残ってしまう。
+
+これは、テストでは通るけれど、メンテナとしては気になる変更です。
+
+抽象を作ったなら、その抽象を使い切ってほしい。既存の設計意図があるなら、その意図に沿ってほしい。今たまたま動くことだけではなく、次に触る人がどの前提で読めばよいのかを残してほしい。
+
+この感覚は、コーディングエージェントと一緒に作業するときにも必要になります。
+
+エージェントに「作って」と頼むだけなら、こちらは利用者に近い立場です。けれど、出てきた差分を受け入れ、将来もそのコードを育てるなら、こちらは少しメンテナになります。
+
+大きなOSSのメンテナでなくても、自分のリポジトリには自分なりの文脈があります。
+
+どこまでを今の変更に含めるのか。
+どの抽象を使うのか。
+どのテストで守るのか。
+どの違和感は今直し、どの違和感は別のissueに分けるのか。
+
+こうした判断は、エージェントが完全には肩代わりできません。むしろ、エージェントが実装できる範囲が広がるほど、人間の側に残る判断が見えやすくなります。
+
+### 評価を一度で終わらせない
+
+ここで、単に「依頼文を丁寧に書きましょう」とまとめてしまうと、少し足りない気がします。
+
+目的を書く。成功条件を書く。触ってよい範囲を書く。そういう心がけは大事ですし、Harness Engineeringの文脈でもよく語られています。実際、エージェントに渡す情報が曖昧なら、出てくる差分も曖昧になります。
+
+でも、FrontierCodeの記事でおもしろかったのは、依頼の書き方そのものよりも、評価を作ったあとにもう一度それを疑っているところでした。
+
+タスク作者は、rubricを書いて終わりにはしていません。あえて雑な解法やズルい解法を作り、その評価で通ってしまわないかを見る。逆に、正しい別解が不当に落ちないかも見る。記事では、そうした検証のためにhack reportを作ると説明されています。
+
+これは、エージェントを使う人間にもかなり大事な視点だと思います。
+
+良い依頼文を書くことは、入口を整えることです。
+
+一方で、良い評価を育てることは、出口を整えることです。
+
+エージェントの出力は、最初に書いた条件の隙間を通ってくることがあります。テストは通っているけれど、期待した抽象には乗っていない。変更範囲は小さいけれど、将来の変更点を増やしている。仕様は満たしているけれど、レビューで説明しづらい形になっている。
+
+そういう差分を見たときに、「プロンプトが悪かった」で終わらせない。
+
+この差分を受け入れたくない理由は何だったのか。最初の依頼に書けるほど明確だったのか。それとも、差分を見て初めて自分の中の判断が見えたのか。
+
+そこを次の依頼やレビュー基準に戻していく。
+
+この往復が、FrontierCodeから持ち帰れる新しいところだと思います。
+
+### 価値判断もループに入れる
+
+コーディングエージェントを使っていると、こちらの判断も少しずつ試されます。
+
+AIが書いたから不安なのではありません。人間が書いたコードでも、よくわからない差分はあります。逆に、エージェントが書いた差分でも、既存の文脈をよく読んでいて、かなり気持ちよくmergeできることもあります。
+
+だから問題は、誰が書いたかではなく、何を根拠に受け入れるかです。
+
+最近のCoding Agentには、`/loop`や`/goal`のように、実行を一回きりで終わらせないための仕組みがあります。計画し、実装し、検証し、失敗したら戻る。そういうループをどう設計するかが、エージェントを使う側にも求められています。
+
+ただ、ループに入るべきなのは、実行手順だけではないのかもしれません。
+
+私たちの価値判断も、たぶんループに入る必要があります。
+
+最初に「この変更はこうあるべき」と思っていても、実際の差分を見ると、別の違和感が見えてくることがあります。逆に、最初は気にしていた制約が、実装してみるとそれほど重要ではなかったとわかることもあります。
+
+FrontierCodeでは、メンテナの判断をrubricに落とし、そのrubric自体を攻撃して、評価として使えるものに近づけています。ここには、判断を固定された正解として置くのではなく、出力とのやり取りの中で更新していく態度があります。
+
+私たちが日常的にやるなら、そこまで大げさでなくてよいと思います。
+
+ただ、エージェントが出した差分に違和感を持ったとき、その違和感を「なんか嫌だ」で終わらせず、少しだけ分解しておく。
+
+これはblockerなのか。
+non-blockerだけれど直したいのか。
+今回の目的から外れているのか。
+既存の抽象を壊しているのか。
+テストが安心の形だけになっているのか。
+
+そうやって、自分が何をmerge可能だと思っているのかを、作業のたびに少しずつ見える形にする。
+
+依頼者が小さなメンテナになる、というのは、最初から完璧な条件を書くことではありません。
+
+出てきた差分を見て、自分の判断を次のループに戻していくことです。
+
+私はしばらく、この感覚を手元に置いておきたいです。エージェントに任せる部分を増やしながら、それでも何をmergeしたいのか、どんな変更なら育てていけるのかは、自分の側にも残しておく。
+
+速く作れることと、受け入れられる形で作れることは、少し違います。
+
+その違いを忘れないための手がかりとして、FrontierCodeの評価設計はとてもよいものに見えました。
